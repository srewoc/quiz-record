# 做题记录系统数据库设计文档

## 1. 文档说明

### 1.1 文档目的
本文档用于明确做题记录系统数据库层的表结构、字段约束、索引策略、扩展组件、迁移建议和实现要点，供后端开发和数据库迁移使用。

### 1.2 技术约束
- 数据库：PostgreSQL
- ORM：SQLAlchemy 2.0
- 迁移工具：Alembic

### 1.3 设计原则
- 满足本期单用户系统需求。
- 优先保证数据一致性、可维护性和后续扩展性。
- 对题目查询和查重场景进行针对性索引设计。
- 对敏感字段采用安全存储方案。

## 2. 数据库对象总览

本期建议至少包含以下数据库对象：

- 扩展：`pg_trgm`
- 表：`questions`
- 表：`llm_configs`

## 3. 扩展设计

### 3.1 pg_trgm 扩展

#### 作用
用于支持题目文本相似度检索，为重复题检测提供候选集。

#### 启用语句
```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
```

#### 使用场景
- `question_text` 相似查询
- `%` 相似匹配或 `similarity()` 排序

## 4. 表设计：questions

### 4.1 表用途
存储题目主数据，包括文本、科目分类、错题标志、图片地址和参考答案。

### 4.2 建表建议

```sql
CREATE TABLE questions (
    id BIGSERIAL PRIMARY KEY,
    question_text TEXT NOT NULL,
    main_subject VARCHAR(32) NOT NULL,
    sub_subject VARCHAR(64) NOT NULL,
    is_wrong BOOLEAN NOT NULL DEFAULT FALSE,
    image_url TEXT NULL,
    reference_answer TEXT NULL,
    is_deleted BOOLEAN NOT NULL DEFAULT FALSE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### 4.3 字段定义

| 字段名 | 类型 | 允许空 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | bigint | 否 | 自增 | 主键 |
| question_text | text | 否 | - | 题目文本 |
| main_subject | varchar(32) | 否 | - | 主科目 |
| sub_subject | varchar(64) | 否 | - | 子科目 |
| is_wrong | boolean | 否 | false | 是否错题 |
| image_url | text | 是 | null | 题目图片 URL |
| reference_answer | text | 是 | null | Markdown 格式参考答案 |
| is_deleted | boolean | 否 | false | 逻辑删除标记 |
| created_at | timestamptz | 否 | now() | 创建时间 |
| updated_at | timestamptz | 否 | now() | 更新时间 |

### 4.4 字段约束

#### 4.4.1 主科目约束
允许值：
- `数学`
- `408`

#### 4.4.2 子科目约束
允许值：
- `数学-高数`
- `数学-线代`
- `数学-概率论`
- `408-数据结构`
- `408-计组`
- `408-操作系统`
- `408-计网`

#### 4.4.3 主子科目一致性约束
建议通过 `CHECK` 约束保证主科目和子科目归属一致：

```sql
CHECK (
    (main_subject = '数学' AND sub_subject IN ('数学-高数', '数学-线代', '数学-概率论'))
    OR
    (main_subject = '408' AND sub_subject IN ('408-数据结构', '408-计组', '408-操作系统', '408-计网'))
)
```

#### 4.4.4 逻辑删除约束
- 逻辑删除不物理删行。
- 所有默认业务查询必须附带 `is_deleted = false` 条件。

### 4.5 索引设计

#### 4.5.1 题目文本相似检索索引
```sql
CREATE INDEX idx_questions_question_text_trgm
ON questions
USING gin (question_text gin_trgm_ops)
WHERE is_deleted = false;
```

#### 4.5.2 列表排序索引
```sql
CREATE INDEX idx_questions_updated_at
ON questions (updated_at DESC)
WHERE is_deleted = false;
```

#### 4.5.3 常用筛选索引
```sql
CREATE INDEX idx_questions_main_subject
ON questions (main_subject)
WHERE is_deleted = false;

CREATE INDEX idx_questions_sub_subject
ON questions (sub_subject)
WHERE is_deleted = false;

CREATE INDEX idx_questions_is_wrong
ON questions (is_wrong)
WHERE is_deleted = false;
```

### 4.6 查询建议

#### 4.6.1 列表查询
```sql
SELECT *
FROM questions
WHERE is_deleted = false
ORDER BY updated_at DESC
LIMIT :page_size OFFSET :offset;
```

#### 4.6.2 查重候选查询
```sql
SELECT id, question_text, main_subject, sub_subject, is_wrong, image_url, reference_answer, created_at, updated_at
FROM questions
WHERE is_deleted = false
ORDER BY similarity(question_text, :input_text) DESC
LIMIT 20;
```

## 5. 表设计：llm_configs

### 5.1 表用途
存储系统可用的 LLM 连接配置，用于统一 LLM 调用模块读取当前启用配置。

### 5.2 建表建议

```sql
CREATE TABLE llm_configs (
    id BIGSERIAL PRIMARY KEY,
    config_name VARCHAR(128) NOT NULL,
    provider_type VARCHAR(32) NOT NULL,
    base_url TEXT NOT NULL,
    api_key_encrypted TEXT NOT NULL,
    model_name VARCHAR(128) NOT NULL,
    is_active BOOLEAN NOT NULL DEFAULT FALSE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### 5.3 字段定义

| 字段名 | 类型 | 允许空 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | bigint | 否 | 自增 | 主键 |
| config_name | varchar(128) | 否 | - | 配置名称 |
| provider_type | varchar(32) | 否 | - | 供应商类型，本期固定为 openai |
| base_url | text | 否 | - | OpenAI 兼容服务地址 |
| api_key_encrypted | text | 否 | - | 加密存储的 API Key |
| model_name | varchar(128) | 否 | - | 模型名称 |
| is_active | boolean | 否 | false | 是否启用 |
| created_at | timestamptz | 否 | now() | 创建时间 |
| updated_at | timestamptz | 否 | now() | 更新时间 |

### 5.4 字段约束

#### 5.4.1 供应商约束
本期仅允许：
- `openai`

建议 `CHECK` 约束：

```sql
CHECK (provider_type IN ('openai'))
```

#### 5.4.2 启用状态唯一约束
同一时刻仅允许一个启用配置。

建议使用部分唯一索引：

```sql
CREATE UNIQUE INDEX uq_llm_configs_single_active
ON llm_configs ((is_active))
WHERE is_active = true;
```

说明：
- 该索引可确保最多只有一条记录满足 `is_active = true`
- 启用切换时应放在同一事务内完成

### 5.5 索引设计

```sql
CREATE INDEX idx_llm_configs_updated_at
ON llm_configs (updated_at DESC);

CREATE INDEX idx_llm_configs_provider_type
ON llm_configs (provider_type);
```

### 5.6 安全存储要求
- `api_key_encrypted` 不得保存明文 API Key。
- 推荐在应用层进行加密后落库。
- 若后续采用密钥管理服务，可将该字段改为外部密钥引用。
- API Key 不允许通过普通查询接口原样返回。

## 6. 时间字段与自动更新策略

### 6.1 created_at
- 插入时自动写入当前时间。
- 创建后不再修改。

### 6.2 updated_at
- 每次更新记录时自动刷新。
- 可在 SQLAlchemy ORM 层通过 `onupdate` 处理。

### 6.3 SQLAlchemy 建议
```python
from sqlalchemy import DateTime, func
from sqlalchemy.orm import mapped_column

created_at = mapped_column(DateTime(timezone=True), server_default=func.now(), nullable=False)
updated_at = mapped_column(
    DateTime(timezone=True),
    server_default=func.now(),
    onupdate=func.now(),
    nullable=False,
)
```

## 8. Alembic 迁移建议

### 8.1 首次迁移内容
- 创建 `questions` 表
- 创建 `llm_configs` 表
- 启用 `pg_trgm` 扩展
- 创建相关索引和约束

### 8.2 迁移顺序建议
1. 启用 `pg_trgm`
2. 创建基础表结构
3. 创建约束
4. 创建索引

### 8.3 回滚建议
- 回滚时先删除索引
- 再删除约束
- 最后删除表
- `pg_trgm` 扩展通常可保留，不建议在回滚中移除

## 9. 数据一致性策略

### 9.1 题目数据
- 应用层和数据库层双重保证主子科目一致。
- 删除采用逻辑删除，避免数据误删。

### 9.2 LLM 配置数据
- 启用配置切换必须在事务内完成。
- 当新配置设为启用时，其他记录需同步置为未启用。
- 若更新失败，应回滚，保持原启用状态。

## 10. 性能与容量建议

### 10.1 questions 表
- 题目文本较长，建议使用 `TEXT` 类型。
- 相似检索依赖 `pg_trgm`，数据量增长后需定期关注索引大小和查询计划。

### 10.2 llm_configs 表
- 数据量通常很小，查询压力低。
- 重点在于安全性和唯一启用约束，而非查询性能。

## 11. 测试与验证清单

### 11.1 questions 表
- 插入题目时必填字段是否生效
- 主科目与子科目不一致时是否被阻断
- 逻辑删除后默认查询是否被过滤
- `updated_at` 是否在更新后刷新
- `pg_trgm` 索引是否可被相似查询命中

### 11.2 llm_configs 表
- API Key 是否以密文方式存储
- 同时插入两个 `is_active = true` 是否被唯一约束阻止
- 激活切换事务是否能正确保持唯一启用
- 普通查询是否不会返回明文 API Key

## 12. 落地建议
- 表名使用复数形式：`questions`、`llm_configs`
- 使用 SQLAlchemy 枚举校验配合数据库 `CHECK` 约束双重兜底
- 将“逻辑删除过滤”收敛到 repository 或 service 层，避免遗漏
- 将 LLM 配置解密逻辑集中在统一 LLM 模块中，不在路由层直接处理
