# 做题记录系统需求文档

## 1. 文档概述

### 1.1 文档名称
做题记录系统需求文档

### 1.2 文档目的
本文档用于明确做题记录系统的业务目标、功能范围、页面交互、数据结构、接口边界、非功能要求与验收标准，作为产品设计、前后端开发、数据库设计、测试验收和后续迭代的统一依据。

### 1.3 适用范围
本需求文档适用于以下技术栈下的系统建设：

- 前端：Next.js + TypeScript
- 后端：FastAPI
- 数据库：PostgreSQL
- ORM：SQLAlchemy 2.0
- 迁移：Alembic

### 1.4 读者对象
- 产品设计人员
- 前端开发人员
- 后端开发人员
- 测试人员
- 项目维护人员

## 2. 项目背景与目标

### 2.1 项目背景
用户在日常刷题过程中，需要长期沉淀题目、整理错题、避免重复录入、保留题目图片和参考答案，并支持后续复习和检索。现需建设一个单用户 Web 做题记录系统，用于统一管理题目数据，并借助 OCR、PostgreSQL 相似匹配和 LLM 能力提升录题与查重效率。

### 2.2 项目目标
- 支持用户集中记录做题内容，沉淀个人题库。
- 支持错题标记，便于后续复习。
- 支持通过文本和图片两种方式检索题目。
- 支持新增、修改、删除题目，并展示 Markdown 格式参考答案。
- 支持 OCR 自动识别题目文本，减少手动录入成本。
- 支持基于 `pg_trgm` 和 LLM 的题目查重，减少重复数据。
- 支持主科目和子科目的自动识别与人工修正。
- 支持通过统一的 LLM 调用模块集中管理模型配置与调用。

### 2.3 成功标准
- 用户可稳定完成题目的新增、查询、修改、删除和答案查看。
- 重复题在新增环节可被有效拦截。
- 图片录题可完成 OCR 识别并驱动后续查重和科目识别。
- 删除操作会真实移除数据库记录，删除后同题干可重新新增。

## 3. 术语定义

### 3.1 题目
系统中的核心记录对象，包含题目文本、科目分类、错题标识、题目图片 URL 和参考答案等信息。

### 3.2 主科目
题目的一级分类，本期固定枚举为：

- 数学
- 408

### 3.3 子科目
题目的二级分类，本期固定枚举为：

- 数学：高数、线代、概率论
- 408：数据结构、计组、操作系统、计网

### 3.4 OCR 模块
接收题目图片并输出固定 Markdown 格式题目文本的识别模块。

### 3.5 题目匹配与科目识别模块
接收输入题目文本和数据库候选题目列表，输出是否已存在、匹配题目 ID 以及主科目、子科目等 JSON 结果的 LLM 模块。

### 3.6 删除
删除时从数据库中真实移除题目记录。删除后默认列表、详情和查询结果均不再返回该题目，同题干可重新新增为新记录。

### 3.7 LLM 管理页面
用于配置系统使用的 LLM 服务连接信息的管理页面，用户可维护模型调用所需的 URL、API Key、模型名称和启用状态等参数。

### 3.8 统一 LLM 调用模块
后端内部统一封装的 LLM 调用能力模块，负责读取当前启用的 LLM 配置并对外提供标准化调用接口，供题目匹配与科目识别等业务模块复用。

## 4. 用户与系统边界

### 4.1 用户角色
本系统为单用户系统，仅有一个使用者角色，不引入账号体系、角色权限和多租户机制。

### 4.2 系统边界
本期纳入需求范围的内容包括：

- 题目列表分页展示
- 文本查询与图片查询
- OCR 识别
- 题目查重
- 科目识别
- LLM 管理页面
- 统一 LLM 调用模块
- 新增题目
- 修改题目
- 删除题目
- Markdown 参考答案展示与编辑

本期不纳入需求范围的内容包括：

- 用户注册、登录、权限控制
- 多用户数据隔离
- 收藏、归档、标签系统
- 批量导入导出
- 题目版本管理
- 自动生成答案
- 操作审批流

## 5. 业务规则总览

### 5.1 基本规则
- 每条题目记录必须包含题目文本、主科目、子科目和错题标识。
- 主科目和子科目必须满足归属一致性。
- 题目图片 URL 为选填项。
- 参考答案为选填项，内容支持 Markdown 格式。
- 删除采用物理删除。

### 5.2 查重规则
- 新增题目时必须执行重复题检测。
- 重复题检测由 PostgreSQL `pg_trgm` 候选检索和 LLM 判断共同完成。
- 若 LLM 判断题目已存在，则禁止新增。
- 系统应提示题目已存在，并支持用户查看已存在记录。

### 5.3 修改规则
- 修改题目时允许编辑题目文本、主科目、子科目、是否错题、题目图片 URL、参考答案。
- 修改时默认不自动触发 OCR、查重和科目识别流程。
- 提交修改时可执行重复校验提示，但不要求复用新增页面的完整自动识别流程。

### 5.4 删除规则
- 删除行为采用物理删除。
- 删除前需弹出确认提示。
- 删除成功后，题目需从默认列表和默认查询结果中消失。

### 5.5 LLM 配置规则
- 所有业务侧 LLM 调用必须统一通过一个后端模块发起，不允许业务模块各自直连第三方 SDK。
- 用户可在 LLM 管理页面维护 LLM 配置。
- 本期仅支持 OpenAI 库配置。
- 配置项至少包括服务 URL、API Key、模型名称和启用状态。
- 同一时刻只允许一个 LLM 配置处于启用状态。
- API Key 在页面展示时必须脱敏。

## 6. 功能需求

### 6.1 主页：题目检索与管理页

#### 6.1.1 功能目标
用于展示题目数据，并提供统一检索、答案查看、修改和删除操作。

#### 6.1.2 页面组成
- 顶部统一检索区域
- 题目列表区域
- 分页区域

#### 6.1.3 列表展示字段
列表至少展示以下字段：

- 题目文本摘要
- 主科目
- 子科目
- 是否错题
- 题目图片 URL
- 创建时间
- 更新时间
- 操作列

#### 6.1.4 列表默认规则
- 默认仅展示未删除题目。
- 默认按更新时间倒序排列。
- 支持分页查询。

#### 6.1.5 操作按钮
每一行数据后应包含以下操作按钮：

- 答案
- 修改
- 删除

#### 6.1.6 答案查看
- 点击“答案”按钮后，以 Markdown 渲染方式展示参考答案。
- 若参考答案为空，则显示“暂无参考答案”。
- 答案展示区应避免执行脚本内容，防止 XSS。

#### 6.1.7 修改
- 点击“修改”按钮后，进入修改题目页面或修改表单。
- 修改页需加载当前题目完整数据。

#### 6.1.8 删除
- 点击“删除”按钮后，弹出二次确认框。
- 用户确认后执行物理删除。
- 删除成功后提示“删除成功”，并刷新当前列表。
- 删除失败时提示失败原因。

### 6.2 顶部统一检索入口

#### 6.2.1 功能目标
支持用户通过文本或图片查找题目，统一在主页完成查询操作。

#### 6.2.2 检索方式
- 文本查询
- 图片查询

#### 6.2.3 文本查询流程
1. 用户输入题目文本。
2. 系统根据输入文本通过 PostgreSQL `pg_trgm` 在数据库中查找候选题目，返回前 20 条记录。
3. 系统将输入题目文本和候选题目列表发送至 LLM 题目匹配模块。
4. LLM 返回是否存在匹配题目及科目信息。
5. 若存在匹配题目，则展示匹配题目的详细数据。
6. 若不存在匹配题目，则提示“未查询到匹配题目”。

#### 6.2.4 图片查询流程
1. 用户上传题目图片。
2. 系统调用 OCR 模块识别题目文本，输出固定 Markdown 格式文本。
3. 系统根据 OCR 结果通过 PostgreSQL `pg_trgm` 在数据库中查找候选题目，返回前 20 条记录。
4. 系统将 OCR 文本和候选题目列表发送至 LLM 题目匹配模块。
5. LLM 返回是否存在匹配题目及科目信息。
6. 若存在匹配题目，则展示匹配题目的详细数据。
7. 若不存在匹配题目，则提示“未查询到匹配题目”。

#### 6.2.5 检索结果展示规则
- 检索结果仅基于当前数据库中仍存在的题目。
- 若命中题目，需展示题目完整信息，至少包含题目文本、科目、错题标识、图片 URL 和参考答案入口。
- 若服务异常导致检索失败，应提示失败原因并允许用户重新操作。

### 6.3 添加题目页面

#### 6.3.1 功能目标
支持通过图片和手工录入相结合的方式新增题目，并在提交前完成重复题检测和科目识别。

#### 6.3.2 表单字段
- 题目文本
- 主科目
- 子科目
- 是否错题
- 题目图片 URL
- 参考答案

#### 6.3.3 字段规则
- 题目文本：必填，支持多行文本，允许用户编辑 OCR 结果。
- 主科目：必填，允许自动回填和手动选择。
- 子科目：必填，允许自动回填和手动选择。
- 是否错题：必填，布尔值。
- 题目图片 URL：选填，需满足 URL 格式校验。
- 参考答案：选填，支持 Markdown。

#### 6.3.4 新增流程
1. 用户上传一张题目图片。
2. 系统调用 OCR 模块识别出题目文本，并以固定 Markdown 格式回填到题目文本输入框。
3. 系统根据识别出的题目文本在数据库中通过 PostgreSQL `pg_trgm` 检索候选题目，返回前 20 条记录。
4. 系统将输入题目文本和 20 条候选记录输入到题目匹配与科目识别模块。
5. 模块返回 JSON，包含是否存在、命中题目 ID、主科目、子科目和原因说明。
6. 若返回结果为题目已存在，则提示“题目已存在”，禁止继续新增，并支持跳转查看原题。
7. 若返回结果为题目不存在，则将主科目和子科目自动回填到表单。
8. 用户选择题目是否为错题，并可补充或修改题目文本、题目图片 URL 和参考答案。
9. 用户点击“确定”按钮提交表单。
10. 后端校验通过后，将题目写入数据库。

#### 6.3.5 异常处理
- OCR 失败时，系统提示识别失败，并允许用户手动填写题目文本继续新增。
- 查重服务失败时，系统提示“未完成重复校验”，允许用户人工确认后继续录入。
- 科目识别失败时，系统允许用户手动选择主科目和子科目。
- 提交时字段校验失败，应高亮对应字段并给出明确提示。

#### 6.3.6 提交结果
- 新增成功后提示“新增成功”，并跳转至主页或题目详情展示位置。
- 新增失败时保留用户已填写内容，提示失败原因。

### 6.4 修改题目页面

#### 6.4.1 功能目标
支持对既有题目进行编辑维护。

#### 6.4.2 可编辑字段
- 题目文本
- 主科目
- 子科目
- 是否错题
- 题目图片 URL
- 参考答案

#### 6.4.3 修改规则
- 页面打开时展示当前题目已有数据。
- 用户修改后点击保存，系统进行基础字段校验。
- 修改提交时默认不自动触发 OCR 识别。
- 修改提交时默认不自动触发完整查重流程，但可执行重复校验提示。
- 若检测到高相似重复题，可提示用户确认，但不强制复用新增页的阻断流程。

#### 6.4.4 保存结果
- 保存成功后提示“修改成功”，并更新列表显示。
- 保存失败时保留当前表单内容，提示失败原因。

### 6.5 LLM 管理页面

#### 6.5.1 功能目标
支持用户维护系统统一使用的 LLM 连接配置，为题目匹配与科目识别模块提供可复用的模型调用能力。

#### 6.5.2 页面内容
- LLM 配置列表
- 新增配置表单
- 编辑配置表单
- 启用状态展示
- 测试连接按钮

#### 6.5.3 本期支持范围
- 本期仅支持 OpenAI 库相关配置。
- 配置方式按 OpenAI 兼容调用参数设计，便于后续扩展兼容服务。

#### 6.5.4 配置字段
- 配置名称
- 供应商类型
- Base URL
- API Key
- 模型名称
- 是否启用
- 创建时间
- 更新时间

#### 6.5.5 字段规则
- 配置名称：必填，用于区分不同配置。
- 供应商类型：本期固定为 `openai`。
- Base URL：必填，需满足 URL 格式校验。
- API Key：必填，保存时需安全存储，展示时脱敏。
- 模型名称：必填，例如 `gpt-4o-mini` 或其他 OpenAI 兼容模型名。
- 是否启用：布尔值；若当前配置被启用，则系统业务调用使用该配置。

#### 6.5.6 页面操作
- 支持新增 LLM 配置。
- 支持编辑已有 LLM 配置。
- 支持查看配置基础信息，API Key 仅显示脱敏值。
- 支持启用某个配置；启用新配置时，原启用配置自动失效。
- 支持测试连接，验证当前配置是否可用于模型调用。

#### 6.5.7 业务规则
- 若系统没有可用的启用配置，则依赖 LLM 的业务功能应提示“LLM 未配置”。
- 题目匹配与科目识别模块调用前，需先读取当前启用配置。
- 测试连接仅用于验证配置可用性，不写入业务数据。
- 本期不强制要求删除配置能力；若实现删除，则不得删除当前唯一启用配置。

#### 6.5.8 异常处理
- Base URL 格式错误时，提交失败并提示。
- API Key 为空时，提交失败并提示。
- 测试连接失败时，提示失败原因，不影响配置保存。
- 启用配置失败时，需提示失败原因并保持原启用状态不变。

### 6.6 删除题目

#### 6.6.1 功能目标
支持用户对题目执行物理删除。

#### 6.6.2 删除流程
1. 用户在列表中点击“删除”按钮。
2. 系统弹出确认框。
3. 用户确认后，后端从数据库中删除该题目记录。
4. 删除成功后刷新列表，题目不再显示。

#### 6.6.3 删除限制
- 已删除题目不应出现在默认列表中。
- 已删除题目不应作为检索结果返回。

### 6.7 参考答案管理

#### 6.7.1 功能目标
支持参考答案的录入、编辑和展示。

#### 6.7.2 业务规则
- 参考答案支持 Markdown 格式存储。
- 新增和修改题目时均可编辑参考答案。
- 查看答案时按 Markdown 渲染。
- 渲染过程中必须进行安全处理，不允许执行恶意脚本。

## 7. 模块需求

### 7.1 OCR 模块

#### 7.1.1 输入
- 题目图片文件或图片内容

#### 7.1.2 输出
- 固定 Markdown 格式的题目文本

#### 7.1.3 功能要求
- 能识别常见题目图片中的文本内容。
- 输出应尽量保留题干结构和排版语义。
- 输出内容用于后续查重和表单回填。

#### 7.1.4 失败处理
- OCR 失败时返回明确错误信息。
- 不因 OCR 失败阻断用户手工录题。

### 7.2 题目匹配与科目识别模块

#### 7.2.1 输入
- 当前输入题目文本
- 数据库候选题目集合

#### 7.2.2 输出
输出为 JSON，对外至少包含以下字段：

```json
{
  "exists": true,
  "matched_question_id": 123,
  "main_subject": "数学",
  "sub_subject": "高数",
  "reason": "与候选题目语义一致，判定为重复题"
}
```

#### 7.2.3 功能要求
- 根据输入题目文本和候选题目集合，判断题目是否已存在。
- 在题目不存在时识别主科目和子科目。
- 输出格式必须稳定，便于后端解析。

#### 7.2.4 主科目枚举
- 数学
- 408

#### 7.2.5 子科目枚举
- 高数
- 线代
- 概率论
- 数据结构
- 计组
- 操作系统
- 计网

#### 7.2.6 失败处理
- 模块调用失败时返回错误信息。
- 调用失败不应导致新增页面完全不可用，应允许用户手动补全信息后提交。

### 7.3 统一 LLM 调用模块

#### 7.3.1 模块目标
为所有业务侧 LLM 调用提供统一入口，集中处理配置读取、客户端初始化、调用参数封装、异常处理和日志记录。

#### 7.3.2 模块职责
- 读取当前启用的 LLM 配置。
- 基于配置初始化 OpenAI 客户端。
- 对外暴露统一调用方法，供题目匹配与科目识别等模块调用。
- 统一处理超时、认证失败、网络失败和返回格式异常。
- 统一记录调用日志和错误日志。

#### 7.3.3 本期支持
- 本期仅支持 OpenAI 库配置。
- 模块内部应预留后续扩展其他供应商的能力，但本期无需实现其他供应商适配。

#### 7.3.4 调用规则
- 题目匹配与科目识别模块不得直接持有第三方 SDK 配置。
- 所有业务模块必须通过统一 LLM 调用模块发起请求。
- 若未找到启用配置，模块返回明确错误，由上层业务决定提示文案。

#### 7.3.5 日志要求
- 记录调用时间、调用模块、使用的配置 ID、调用结果和失败原因。
- 不得在日志中明文输出 API Key。

### 7.4 LLM 配置管理模块

#### 7.4.1 模块目标
提供 LLM 配置的新增、编辑、启用状态切换、读取和测试连接能力。

#### 7.4.2 功能职责
- 保存和更新配置数据。
- 保证仅有一个配置处于启用状态。
- 提供读取当前启用配置的能力。
- 提供测试连接能力。

#### 7.4.3 安全要求
- API Key 必须安全存储。
- API Key 在接口返回中默认脱敏。
- 非测试连接场景不返回完整密钥内容。

## 8. 数据需求

### 8.1 题目实体字段
题目表至少包含以下字段：

| 字段名 | 类型建议 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| id | bigint / uuid | 是 | 主键 |
| question_text | text | 是 | 题目文本 |
| main_subject | varchar | 是 | 主科目 |
| sub_subject | varchar | 是 | 子科目 |
| is_wrong | boolean | 是 | 是否错题，默认 false |
| image_url | text | 否 | 题目图片 URL |
| reference_answer | text | 否 | Markdown 格式参考答案 |
| created_at | timestamp | 是 | 创建时间 |
| updated_at | timestamp | 是 | 更新时间 |

### 8.2 数据约束
- `question_text` 不能为空。
- `main_subject` 必须属于定义的主科目枚举。
- `sub_subject` 必须属于定义的子科目枚举。
- `sub_subject` 必须与 `main_subject` 的归属一致。
- `image_url` 若不为空，必须满足 URL 格式要求。
- `reference_answer` 支持空值。

### 8.3 索引建议
- 为 `question_text` 配置 `pg_trgm` 相关索引，以支持相似题检索。
- 为 `main_subject`、`sub_subject`、`is_wrong` 建立普通查询索引。
- 为 `updated_at` 建立排序相关索引，以优化默认列表查询。

### 8.4 数据删除规则
- 删除题目时直接移除数据库记录。
- 删除后的题目不参与列表、详情、检索、查重和候选题匹配。

### 8.5 LLM 配置实体字段
LLM 配置表至少包含以下字段：

| 字段名 | 类型建议 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| id | bigint / uuid | 是 | 主键 |
| config_name | varchar | 是 | 配置名称 |
| provider_type | varchar | 是 | 供应商类型，本期固定为 `openai` |
| base_url | text | 是 | OpenAI 兼容服务地址 |
| api_key_encrypted | text | 是 | 加密存储的 API Key |
| model_name | varchar | 是 | 默认调用模型名称 |
| is_active | boolean | 是 | 是否为当前启用配置 |
| created_at | timestamp | 是 | 创建时间 |
| updated_at | timestamp | 是 | 更新时间 |

### 8.6 LLM 配置数据约束
- `config_name` 不能为空。
- `provider_type` 本期固定枚举值为 `openai`。
- `base_url` 必须满足 URL 格式要求。
- `api_key_encrypted` 必须存储加密后的密钥或安全存储引用。
- `model_name` 不能为空。
- 同一时刻仅允许一个配置的 `is_active = true`。

## 9. 接口需求

以下为建议 REST API 边界，实际路径可在详细设计阶段微调，但语义不得偏离本需求。

### 9.1 获取题目列表
- 方法：`GET /questions`
- 用途：分页获取题目列表

请求参数建议：
- `page`
- `page_size`
- `main_subject`
- `sub_subject`
- `is_wrong`
- `keyword`
- `sort`

返回内容至少包含：
- 列表数据
- 总条数
- 当前页码
- 每页条数

### 9.2 获取题目详情
- 方法：`GET /questions/{id}`
- 用途：获取单个题目的完整信息

### 9.3 文本查询
- 方法：`POST /questions/search/text`
- 用途：基于输入文本检索题目

请求体至少包含：
- `question_text`

返回内容至少包含：
- 输入文本
- 候选题目列表
- LLM 匹配结果
- 若命中则返回匹配题目数据

### 9.4 图片查询
- 方法：`POST /questions/search/image`
- 用途：基于图片执行 OCR 后检索题目

请求体至少包含：
- 图片文件或图片内容

返回内容至少包含：
- OCR 识别文本
- 候选题目列表
- LLM 匹配结果
- 若命中则返回匹配题目数据

### 9.5 重复题检测
- 方法：`POST /questions/deduplicate`
- 用途：在新增场景中单独执行查重和科目识别

请求体至少包含：
- `question_text`

返回内容至少包含：
- `exists`
- `matched_question_id`
- `main_subject`
- `sub_subject`
- `reason`

### 9.6 新增题目
- 方法：`POST /questions`
- 用途：新增题目

请求体至少包含：
- `question_text`
- `main_subject`
- `sub_subject`
- `is_wrong`
- `image_url`
- `reference_answer`

### 9.7 修改题目
- 方法：`PUT /questions/{id}`
- 用途：更新题目内容

请求体至少包含：
- `question_text`
- `main_subject`
- `sub_subject`
- `is_wrong`
- `image_url`
- `reference_answer`

### 9.8 删除题目
- 方法：`DELETE /questions/{id}`
- 用途：删除题目

返回结果要求：
- 删除成功返回标准成功响应
- 删除失败返回明确错误信息

### 9.9 获取 LLM 配置列表
- 方法：`GET /llm-configs`
- 用途：获取 LLM 配置列表

返回内容至少包含：
- 配置 ID
- 配置名称
- 供应商类型
- Base URL
- 脱敏后的 API Key
- 模型名称
- 是否启用
- 创建时间
- 更新时间

### 9.10 新增 LLM 配置
- 方法：`POST /llm-configs`
- 用途：新增 LLM 配置

请求体至少包含：
- `config_name`
- `provider_type`
- `base_url`
- `api_key`
- `model_name`
- `is_active`

### 9.11 修改 LLM 配置
- 方法：`PUT /llm-configs/{id}`
- 用途：修改 LLM 配置

请求体至少包含：
- `config_name`
- `provider_type`
- `base_url`
- `api_key`
- `model_name`
- `is_active`

### 9.12 启用 LLM 配置
- 方法：`POST /llm-configs/{id}/activate`
- 用途：将指定配置设为启用状态

返回结果要求：
- 启用成功后，原启用配置自动失效
- 启用失败时返回明确错误信息

### 9.13 测试 LLM 配置连接
- 方法：`POST /llm-configs/test`
- 用途：测试某个配置是否可成功初始化并完成一次最小化模型调用

请求体至少包含：
- `provider_type`
- `base_url`
- `api_key`
- `model_name`

返回内容至少包含：
- 是否成功
- 响应信息或失败原因

## 10. 页面交互与状态要求

### 10.1 通用交互要求
- 所有异步操作均需有加载中状态提示。
- 所有提交按钮在请求处理中应防重复点击。
- 所有成功操作应给出明确成功提示。
- 所有失败操作应给出明确失败提示。

### 10.2 空状态
- 列表无数据时显示空状态提示。
- 答案为空时显示“暂无参考答案”。
- 查询无结果时显示“未查询到匹配题目”。

### 10.3 异常状态
- OCR 调用失败时显示识别失败提示。
- LLM 调用失败时显示识别失败或查重失败提示。
- LLM 未配置时显示“LLM 未配置，请先前往管理页面完成配置”。
- 网络错误时显示请求失败提示，并允许重试。

## 11. 非功能需求

### 11.1 性能要求
- 普通分页列表查询接口目标响应时间不高于 1 秒。
- 文本查询接口目标响应时间不高于 3 秒。
- 图片查询接口由于依赖 OCR 和 LLM，目标响应时间不高于 10 秒。
- OCR 和 LLM 调用属于异步等待型外部处理流程，前端需展示处理中状态。

### 11.2 可用性要求
- 系统核心流程应支持用户独立完成，无需额外培训。
- OCR 或 LLM 失败时应保留人工兜底能力，不中断题目录入主流程。

### 11.3 一致性要求
- 删除后题目不可出现在默认列表和默认检索结果中。
- 修改后题目列表和详情展示的数据必须与数据库最新数据一致。
- 新增成功后应能立即在列表中查询到。

### 11.4 安全要求
- 后端必须对输入字段进行长度和格式校验。
- Markdown 渲染必须进行安全过滤，防止 XSS。
- 图片 URL 仅作字符串存储与展示，不执行远程抓取和下载。
- 所有接口需对非法参数返回明确错误响应。
- LLM API Key 不得以明文形式回传到前端展示页面。
- LLM API Key 应使用加密存储或安全密钥托管方案保存。

### 11.5 日志与审计要求
- 记录 OCR 模块调用结果。
- 记录 LLM 题目匹配与科目识别模块调用结果。
- 记录 LLM 配置新增、修改、启用和测试连接结果。
- 记录新增、修改、删除操作结果。
- 失败场景应记录失败原因，便于排查。

## 12. 验收标准

### 12.1 功能验收
- 新增题目成功：OCR 成功、查重未命中、科目识别成功、提交后题目出现在列表中。
- 重复题阻断：OCR 或文本输入后查重命中，系统提示题目已存在并禁止新增。
- OCR 失败兜底：OCR 失败时，用户可手动录入题目文本并继续提交。
- LLM 科目识别失败兜底：用户可手动选择主科目与子科目并成功保存。
- LLM 配置新增成功：用户可新增一条 OpenAI 配置并在列表中看到脱敏后的配置内容。
- LLM 配置启用成功：启用新配置后，系统实际业务调用切换到该配置。
- LLM 测试连接：输入合法配置后可完成连接测试并返回成功结果。
- 主页分页：默认展示未删除题目，分页切换正常。
- 文本查询：输入题目文本后可返回匹配题目结果或明确无结果提示。
- 图片查询：上传图片后可完成 OCR、候选检索、LLM 判断并展示结果。
- 修改题目：修改题目文本、错题标志、答案后保存成功并正确回显。
- Markdown 答案展示：答案为空时显示空态，答案非空时正确渲染 Markdown。
- 删除题目：执行物理删除后题目从默认列表中消失，数据库记录被移除。

### 12.2 校验验收
- 题目文本为空时提交失败。
- 主科目与子科目不匹配时提交失败。
- 图片 URL 非法时提交失败。
- LLM Base URL 非法时配置提交失败。
- LLM API Key 为空时配置提交失败。
- 非法参数请求时接口返回明确错误信息。

### 12.3 安全验收
- Markdown 内容中的恶意脚本不被执行。
- 已删除数据不会在默认业务查询中暴露。

## 14. 实施说明

### 14.1 前端建议
- 使用 Next.js 页面或 App Router 组织题目列表页、添加页、修改页和 LLM 管理页。
- 使用 TypeScript 类型约束题目实体、查询参数和接口响应。
- Markdown 展示建议使用安全渲染方案，并对 HTML 注入进行过滤。

### 14.2 后端建议
- 使用 FastAPI 提供 REST API。
- 使用 SQLAlchemy 2.0 定义题目实体。
- 使用 Alembic 管理表结构迁移。
- 在 PostgreSQL 中启用 `pg_trgm` 扩展并建立相似匹配索引。
- 将 OpenAI SDK 的调用统一封装到独立服务模块中，业务模块仅依赖统一抽象。

### 14.3 数据库建议
- 题目表命名可采用 `questions`。
- LLM 配置表命名可采用 `llm_configs`。
- 创建时间和更新时间建议由后端统一维护。
